# FINANCE-AGENT 真实业务数据分类与全链路测试规划

> 版本：V1.0
>
> 编制日期：2026-07-14
>
> 数据来源：仓库根目录本地 `数据文件/`（已被 Git 忽略）
>
> 当前任务：先建立可复现、可脱敏、可验收的测试体系，再按批次修复真实数据暴露的问题。

## 1. 目标与边界

本规划用于把真实业务文件逐步接入 FINANCE-AGENT 的文件、Excel、OCR、经营记录、报表、规则审核和老板 AI 助手链路。测试必须回答以下问题：

1. 文件是否能被安全识别、上传、预览、下载和留痕。
2. Excel 是否能选对工作表、表头和数据区，并稳定映射到项目、模板和动态字段。
3. OCR 是否能处理电子 PDF、扫描件、长截图、拍屏表格和多页凭证，并把不确定结果交给人工复核。
4. Excel、OCR、手工补录进入同一套 `BusinessRecord` / `RecordValue` 规则后，金额、日期、方向、来源和状态是否正确。
5. 报表和老板 AI 助手是否只基于已确认结构化数据回答，并与数据库确定性计算一致。
6. 大文件、复杂格式、重复文件、模型故障和并发重试是否会造成泄露、崩溃、重复入账或错误统计。

本轮不做以下事情：

- 不修改、移动、重命名或覆盖 `数据文件/` 中的原始文件。
- 不把原始文件、原始文件名清单、业务明细、人员信息、银行卡、电话、身份证、具体金额或 OCR 原文提交到 Git。
- 不在没有标准答案集的情况下微调模型权重。
- 不把本地文件上传到外部模型、公共对象存储或第三方标注平台。
- 不把汇总表、利润表、日报等派生数据直接当作原始交易重复入账。
- 暂不把“业务级重复”作为自动拒绝条件；第一版先提示并人工确认，待财务明确规则后再启用自动阻断。

## 2. 数据安全红线

1. `数据文件/` 视为只读生产样本，测试程序只允许读取。
2. 所有临时文件放在 Git 忽略的 `.realdata-test/`，原始文件不得进入 `backend/test/fixtures`。
3. 测试数据库必须使用独立名称，例如 `finance_agent_realdata_test`，启动前校验名称以 `_test` 结尾。
4. 测试报告只记录样本别名、SHA-256 前缀、格式、尺寸、错误类别和聚合指标，不记录原始值。
5. 日志、异常、模型调用记录和截图必须脱敏；手机号、证件号、银行卡号、运单号等只保留类型和掩码。
6. 本地模型服务禁止访问外网；模型权重只读挂载，不能复制、转换、删除或提交。
7. 任何确认入库测试都使用测试项目和测试用户，并以唯一 `testRunId` 标记，支持精确清理。
8. 发现数据泄露、非测试数据库写入、原件变化或外部网络传输时立即停止测试。

### 3.3 PDF、图片和 Word 画像

- 23 份 PDF 均能解析：21 份单页、1 份 4 页、1 份 35 页，均未加密。
- 当前 OCR 默认最多 20 页，因此 35 页真实文件会在上传后被 OCR 预处理拒绝。
- 35 张图片分辨率跨度较大，既有小尺寸裁剪，也有最高约 1080x4794 的长截图。
- 视觉抽样确认至少包含：清晰经营日报、拍摄显示器中的 Excel、长电商订单、物流签收/收货信息和物品采购凭证。
- 1 份 DOCX 是大型表格文档，约 166 行；文件模块可接受 DOCX，但当前 Excel/OCR 业务链路不会解析其表格。

## 4. 文件功能与类型分类

测试和实现中统一使用下列匿名数据族，不直接引用原始文件名。

| 数据族 | 业务功能 | 常见格式/结构 | 系统目标路径 | 关键风险 |
| --- | --- | --- | --- | --- |
| `RB-ATT` | 考勤、工时和企业结算 | 多日 Sheet、汇总 Sheet、公式、合并表头 | Excel 导入 -> 人工映射 -> 成本记录 | 同一人员跨日、汇总与明细重复、公式结果 |
| `RB-FRT` | 司机、车辆、揽收、派送和运输账单 | `.xls/.xlsx`、多 Sheet、票数/重量/单价/运费 | Excel 导入 -> 运输费用模板 | 旧格式、宽表、银行卡/人员隐私、金额精度 |
| `RB-EXP` | 报销、备用金、核销和付款申请 | 多月历史 Sheet、隐藏 Sheet、内嵌凭证图片 | Excel 主表 + 文件证据 -> 费用记录 | 文件巨大、历史页误导入、凭证和行关联 |
| `RB-CLM` | 理赔、问题件和赔付 | Excel、Word 表格、公式和说明文本 | 理赔模板 -> 风险规则/审批 | 责任归属、负数/冲销、Word 暂无解析链路 |
| `RB-PAY` | 工资、司机结算和劳务费用 | 汇总/明细 Sheet、个人及银行字段 | 受限模板 -> 成本记录 | 高敏个人数据、权限、汇总与明细双计 |
| `RB-CASH` | 现金流水和资金需求 | 跨月 Sheet、公式、预算与实际 | 财务分析/对账，谨慎入库 | 预算不能当实际、月份边界、余额公式 |
| `RB-MGT` | 利润、经营日报和运营问题 | 月度表、日报截图、汇总指标 | 对账基准、老板 AI 问答基准 | 属于派生数据，禁止再次生成原始交易 |
| `RB-MDM` | 资产、合同、车辆和油耗基础资料 | 主数据表、备注、合同期限 | 后续主数据模块或附件存档 | 当前业务模型不完整，不应强塞进交易表 |
| `RB-EINV` | 电子发票和规范 PDF 票据 | 单页 PDF、可检索文本或标准版式 | OCR -> 人工复核 -> 费用记录 | 发票字段、税额、重复票据 |
| `RB-SCAN` | 扫描凭证、行程和多页收款材料 | 4/35 页 PDF | OCR 分页 -> 人工复核 | 超页、页间归属、扫描质量和处理时长 |
| `RB-SHOT` | 电商订单、物流签收和手机长截图 | JPG/PNG、长图、多项目 | OCR/VL -> 人工选取总额和证据 | 商品价与实付混淆、电话/地址泄露、长图切片 |
| `RB-TABLE-IMG` | 拍屏 Excel 和经营日报图片 | 表格截图、摩尔纹、透视、裁剪 | OCR/VL 表格抽取或仅作为证据 | 行列错位、合计行误识别、低清晰度 |
| `RB-ARC` | 月度归档包 | ZIP 内混合 PDF/JPG/PNG/XLS/XLSX | 安全解包暂存 -> 分类任务 | 路径穿越、压缩炸弹、重复散文件、部分失败 |
| `RB-DUP` | 完全重复和版本副本 | 同哈希、不同文件名、归档/散文件镜像 | 上传提示、幂等和人工判断 | 暂不自动拒绝业务近似重复 |

### 4.1 入账源与对账源必须分开

| 类型 | 示例数据族 | 第一版处理原则 |
| --- | --- | --- |
| 原始交易/成本明细 | `RB-FRT`、`RB-EXP`、`RB-CLM`、经确认的票据 | 可以生成 `BusinessRecord` |
| 劳务/考勤明细 | `RB-ATT`、`RB-PAY` | 选择明细或结算层级，只能确定一个入账粒度 |
| 派生汇总/经营日报 | `RB-MGT` | 只做对账和 AI 测试真值，不直接重复入账 |
| 预算/资金需求 | `RB-CASH` 中的预算 Sheet | 标记为计划数据，不能混入实际收支 |
| 主数据/档案 | `RB-MDM` | 暂存或后续建模，不写入普通费用记录 |

## 5. 当前实现与真实数据的主要差距

| 编号 | 差距 | 影响 | 建议优先级 |
| --- | --- | --- | --- |
| `GAP-01` | 导入器仅支持单个非空 `.xlsx` Sheet、第一行表头 | 多数真实表无法进入映射页 | P1 |
| `GAP-02` | 不支持 `.xls` | 至少 37 份逻辑账单无法上传/导入 | P1 |
| `GAP-03` | 合并单元格直接拒绝，公式行直接报错 | 30/34 含合并，29/34 含公式 | P1 |
| `GAP-04` | 5000 行上限且同步整本加载 | 真实账单最大 30196 行，存在内存和恢复风险 | P1 |
| `GAP-05` | 默认 10 MiB、配置最大 50 MiB | 7 份超过默认值，1 份超过硬上限 | P1 |
| `GAP-06` | 大型工作簿包含 998 个内嵌媒体对象 | 表格解析、凭证抽取和内存控制未分层 | P1 |
| `GAP-07` | OCR 最多 20 页 | 35 页真实 PDF 无法处理 | P1 |
| `GAP-08` | ZIP 和 DOCX 表格没有业务解析路线 | 月度包和 Word 理赔需人工拆分 | P2 |
| `GAP-09` | 真实图片包含长截图、拍屏和复杂表格 | PaddleOCR 的真实准确率未知 | P1 |
| `GAP-10` | 存在完全重复、归档镜像和版本副本 | 可能重复创建任务或记录 | P2，先告警不阻断 |
| `GAP-11` | 汇总表、利润表和日报可能被误当交易 | 会造成经营数据重复和错误利润 | P0 |
| `GAP-12` | 文件含人员、银行、电话和地址等敏感信息 | 日志、截图、模型结果存在泄露风险 | P0 |
| `GAP-13` | 本地模型资产完整但尚未完成真实 GPU 服务验收 | 不能声明 OCR/文本模型达到生产可用 | Gate |

候选改动只有在对应测试先复现失败后才能实施。优先考虑元数据扫描、Sheet/表头选择、导入配置 Profile、流式分块、`.xls` 沙箱转换、长图切片、多页任务拆分和脱敏日志；不要直接为某一个原始文件写死规则。

## 6. 测试环境设计

### 6.1 目录

```text
数据文件/                         # 原始样本，只读，已忽略
.realdata-test/                  # 本地测试工作区，必须忽略
  inventory.local.json          # 原始路径到匿名 sampleId 的本地映射
  labels.local.jsonl            # 人工真值，只保存在本机
  derived/                      # 旋转、裁剪、压缩、分页等派生样本
  reports/                      # 含脱敏明细的本地报告
backend/test/fixtures/generated/ # 仅允许提交无业务信息的合成回归样本
docs/                            # 只提交聚合结论和验收报告
```

### 6.2 数据库与服务

- PostgreSQL：独立 `finance_agent_realdata_test`。
- 后端：测试专用 `.env.test.local`，不得复用生产数据库或上传目录。
- 文件目录：独立测试隔离区和存储区，执行前记录剩余磁盘。
- ClamAV：安全测试必须使用 fail-closed 模式；本地功能基线可另跑 basic，但不能替代生产门禁验收。
- AI：先跑 Mock 确定性回归，再启用本地 Qwen 文本服务。
- OCR：先跑 Mock 状态机回归，再启用 PaddleOCR-VL；Qwen-VL 只用于明确的低置信度/复杂表格样本。
- 浏览器：Playwright 使用独立端口和测试账号，不保存原始业务截图到仓库。

### 6.3 样本标识

每个文件生成匿名 ID，例如 `RB-FRT-XLS-001`。本地清单至少保存：

```json
{
  "sampleId": "RB-FRT-XLS-001",
  "sha256": "full-local-hash",
  "duplicateGroup": "DUP-003",
  "family": "RB-FRT",
  "format": "xls",
  "sizeBytes": 0,
  "sensitivity": "restricted",
  "expectedRoute": "excel-import",
  "split": "calibration"
}
```

跟踪文档中只能出现 `sampleId` 和聚合数据，不能出现 `full-local-hash`、原始路径或真值。

## 7. 标准答案集

### 7.1 三层真值

| 层级 | 生成方式 | 用途 |
| --- | --- | --- |
| L1 文件结构真值 | 程序只读生成：哈希、格式、页数、Sheet、行列、隐藏/公式/合并/媒体 | 文件安全和解析器测试 |
| L2 字段真值 | 财务人员对抽样文件人工标注并复核 | OCR、映射和动态字段准确率 |
| L3 会计真值 | 财务确认入账粒度、收支方向、项目、期间、应计总额和可作废规则 | 记录、报表和 AI 数字验收 |

### 7.2 抽样策略

1. 每个数据族至少选 5 个代表样本；样本不足时全选。
2. 清晰电子票据、普通截图、长截图、拍屏、4 页 PDF、35 页 PDF都必须覆盖。
3. 每种 Excel 复杂性至少覆盖：单 Sheet、多 Sheet、隐藏 Sheet、合并表头、公式、内嵌图片、宽表、大于 5000 行、`.xls`。
4. 6 组完全重复文件和 3 组 ZIP/散文件镜像全部进入鲁棒性集，不进入准确率分母。
5. 相同哈希、同一业务对象不同副本和同一月份连续版本必须放在同一数据分区，防止泄漏。
6. 初始建议按 60% 校准、20% 验证、20% 最终盲测拆分；最终盲测集在阈值确定前不用于调 Prompt。

### 7.3 人工标注字段

通用字段：文档类型、业务日期、项目、模板、收支方向、金额、币种、费用类别、供应商/付款对象、说明、来源页码、是否允许入账。

运输字段：司机、车辆、票数、重量、单价、线路、揽收/派送类型。

劳务字段：人员、岗位、工时、出勤、计费单价、结算金额。

票据字段：发票/票据类型、号码、开票日期、含税金额、税额、销方、购买内容。

隐私字段只在本地加密真值文件中保存；测试报告只输出“正确/错误/缺失/低置信度”。

## 8. 执行批次

### B0：冻结清单与兼容性基线

目标：不调用业务写接口，只确定每个样本当前会走哪条路线、在哪里失败。

工作项：

- 生成匿名清单、全量哈希、重复组和格式签名。
- 扫描 Excel Sheet、隐藏状态、维度、公式、合并、媒体和压缩展开大小。
- 扫描 PDF 页数、加密和活动内容；扫描图片尺寸、方向和长宽比。
- 对当前文件安全服务、Excel 解析器、OCR 预处理器做 dry-run 兼容性矩阵。
- 错误必须归为 `supported`、`needs-profile`、`needs-conversion`、`manual-only`、`security-rejected`，不能只记“失败”。

完成门禁：原件哈希前后不变；没有业务数据写入；报告不含原始值；所有 112 个物理文件都有匿名分类。

### B1：文件安全与上传边界

目标：保证不支持的真实文件被清晰拒绝，支持的文件不会因尺寸、扫描或权限产生不一致状态。

重点用例：

- 9.99 MiB、10 MiB、10 MiB+1、接近 50 MiB、50 MiB、50 MiB+1。
- 实际 7 个大于默认限制的工作簿和 1 个超过 50 MiB 的工作簿。
- `.xls`、`.zip`、`.docx` 的显式路由或统一 4xx，不允许 500。
- OOXML 展开大小、压缩比、宏、外链、嵌入对象和路径穿越。
- ClamAV 不可用、超时、感染、扫描中和扫描失败均 fail closed。
- 相同哈希重复上传、不同文件名重复上传、ZIP 与散文件重复上传。
- finance、boss、employee、reviewer 的上传、预览、下载和作废权限。
- 日志、审计和错误响应中无原始路径和敏感内容。

完成门禁：支持/拒绝政策文档化；无 500；无未扫描可读文件；无原件泄露；配额与磁盘水位可恢复。

### B2：Excel 真实适配

目标：形成“扫描 -> 选 Sheet/数据区 -> 预览 -> 映射 -> 校验 -> 人工确认 -> 分块入库”的可恢复流程。

重点用例：

- 多 Sheet 文件展示完整目录，默认不导入隐藏 Sheet。
- 用户显式选择明细 Sheet；汇总 Sheet 只作为对账，不与明细同时入账。
- 自动探测候选表头行，但最终选择可人工修改。
- 合并表头展开后保持分组语义；无法确定时停止并提示。
- 公式不在服务端执行。可读取可信缓存结果时同时保留公式来源标记；缺少缓存或结果异常时要求人工处理。
- `.xls` 在隔离工作进程中解析或转换，不依赖服务器桌面 Excel/COM。
- 内嵌媒体不进入普通行解析内存；如需作为凭证，单独建立附件清单和关联确认。
- 4999、5000、5001 和真实 30196 行场景使用分块、lease、进度、重试和断点恢复。
- 180 列宽表、空列、重复列名、多行标题、合计行、空白行、隐藏行列。
- Excel 日期序列、文本日期、1900/1904 日期系统、千分位、小数、负数、括号负数和空值。
- 字段建议只能创建待审核建议，不得自动改变模板或数据库列。
- 解析/映射阶段不生成经营记录；确认时事务写入，失败可重试且不重复。

完成门禁：每个支持的数据族至少 3 份真实样本端到端通过；金额和日期与 L2/L3 真值一致；中断重试不重复；所有关键动作有 audit/ledger。

### B3：OCR 与视觉模型基线

目标：用 PaddleOCR-VL 建立真实准确率基线，并验证 Qwen-VL 只在必要时按需复核。

重点用例：

- 单页电子 PDF、普通照片、手机长截图、拍屏 Excel、经营日报表格、4 页和 35 页 PDF。
- 90/180/270 度旋转、透视、裁剪、模糊、压缩、摩尔纹、阴影和超长图切片。
- 文档分类后再抽字段，不能用同一票据 Prompt 处理所有图片。
- 电商订单区分商品单价、小计、优惠、运费和最终实付。
- 物流截图中的电话、地址、运单号默认脱敏，不作为普通 AI 回答上下文。
- 表格 OCR 验证行列绑定，不能只测全文字符识别。
- 35 页 PDF采用页范围、拆分任务或批处理，不允许静默截断前 20 页。
- 低置信度、字段冲突和无法归属的页必须进入人工复核。
- OCR 原文、字段、置信度、页码、框选、模型版本和尝试记录完整保留。
- 未确认任务绝不生成 `BusinessRecord`；人工修正写 correction、audit 和 ledger。
- OCR 超时、OOM、Provider 5xx、进程重启和队列恢复不影响普通业务功能。

完成门禁：模型指标达到第 10 节的最低目标，或明确降级为“人工辅助模式”；无自动入账；无隐私泄露；连续运行 30 分钟无 OOM/重启。

### B4：经营记录与会计一致性

目标：验证 Excel、OCR、手工补录和工单终审进入同一套会计规则。

重点用例：

- 项目、模板和字段归属校验；停用/归档对象不能新增记录。
- 收入、成本和费用方向由后端模板/政策决定，不能信任前端传值。
- 金额使用 Decimal 字符串契约，汇总无浮点误差。
- 原始交易、结算汇总、经营日报和预算数据分层，禁止重复计入实际经营数据。
- 草稿、确认、作废、并发确认和重试的状态机与幂等。
- 记录保留来源文件、来源任务、项目/模板版本和确认快照。
- 创建、确认、修正、作废均有 audit；影响经营数据的动作均有 ledger。

完成门禁：L3 抽样逐条一致；所有项目/月度汇总可回溯到记录；重复重试只产生一个确认结果。

### B5：报表、规则与老板 AI 助手

目标：让派生报表成为确定性真值，文本模型只负责理解问题和组织工具结果。

重点用例：

- 财务日/周/月、老板日报、项目月报只统计 confirmed 且未作废记录。
- 北京时间日/月边界、跨月记录、负数冲销、零金额和空类别。
- 用 `RB-MGT` 汇总表做对账，不把它再次导入为交易。
- 风险规则覆盖异常大额、缺附件、字段冲突、重复提示和跨期异常。
- 建立 50 至 100 个老板问题，覆盖总额、同比/环比、项目排行、成本结构、异常、下钻、无数据和越权。
- 数字答案必须来自后端工具/SQL，模型不得自行计算或猜测。
- 回答携带期间、口径和数据来源；无数据时明确说明，不生成虚构数字。
- OCR 原文、Excel 单元格和文件名中的“忽略规则”等文本均按不可信数据处理，测试 Prompt Injection。
- employee、finance、reviewer 不能通过 AI 工具读取 boss-only 或他人敏感数据。

完成门禁：数字答案与报表工具 100% 一致；越权拒绝 100%；无数据不编造 100%；工具选择准确率达到目标。

### B6：性能、故障恢复和长稳

目标：为真实文件量建立资源基线，而不是单纯提高限制。

测试维度：

- 文件：1 MiB、10 MiB、接近 50 MiB、超过 50 MiB、含 248 个内嵌媒体对象。
- Excel：4999/5000/5001/30196 行，普通列和 180 列宽表。
- OCR：单页、4 页、20 页、21 页、35 页，普通图和 1080x4794 长图。
- 并发：1/3/5 个上传与导入；OCR 初始单并发排队；文本问答与 OCR 同时运行。
- 故障：后端重启、模型重启、数据库短断、ClamAV 不可用、磁盘水位不足、任务 lease 过期。
- 指标：上传吞吐、解析耗时、确认耗时、RSS/堆峰值、数据库写入量、GPU 显存、首 Token、每页 OCR 耗时、队列等待和恢复时间。

完成门禁：无 OOM、无进程崩溃、无孤儿临时文件、无重复确认；P95 在配置超时内。首轮先记录基线，再由实测结果和业务等待时间确定正式 SLO。

### B7：财务 UAT 与最终回归

目标：由财务人员验证入账粒度、字段映射、重复提示和人工复核体验。

UAT 场景：

1. 选择一个运输账单完成 Excel 导入和对账。
2. 选择一个考勤结算表，明确明细/汇总入账政策。
3. 选择一个报销工作簿，验证主表与内嵌凭证的关系。
4. 选择电子票据、长订单截图和多页凭证完成 OCR 纠错和确认。
5. 验证报表与人工汇总一致。
6. 老板使用标准问题集提问并核对数字、口径和来源。
7. 重复上传相同文件时出现提示，但由财务决定继续或取消。

完成门禁：财务签署字段和入账规则；自动化回归、构建、安全检查全部通过；未解决项有明确负责人和降级方案。

## 11. 问题发现与修改规则

每个问题按以下顺序处理：

1. 用匿名 `sampleId` 记录复现步骤、预期、实际、日志关联 ID 和影响范围。
2. 判断是格式兼容、业务规则、数据质量、模型准确率、性能、安全还是产品交互问题。
3. 先增加最小失败测试；无法提交真实样本时，生成不含业务信息的等价合成 fixture。
4. 优先修通用解析器、策略或 Provider，不按原始文件名、客户名或月份写分支。
5. 修改后运行单测、真实 PostgreSQL 集成测试、相关 Playwright、构建和仓库卫生检查。
6. 对金额、权限、确认、作废、文件和模型调用检查 audit/ledger/日志。
7. 更新兼容矩阵和聚合报告；原始错误值继续留在本地报告，不进入 Git。

严重级别：

| 级别 | 定义 | 处理 |
| --- | --- | --- |
| P0 | 数据泄露、越权、错误入账、报表双计、写入非测试数据库 | 立即停止，修复后全量回归 |
| P1 | 主数据族无法导入、任务不可恢复、模型错误高置信度、OOM | 阻塞对应功能验收 |
| P2 | 少数格式需人工整理、重复提示不完善、体验问题 | 可带降级方案进入 UAT |
| P3 | 文案、非关键展示和优化建议 | 记录后排期 |

重复文件当前策略：记录哈希和候选重复关系、在确认前提示、测试幂等；不在财务给出“哪些字段构成同一业务”的规则前自动拒绝业务近似重复。

## 12. 总验收门禁

### G0 数据与隐私

- 原始 112 个物理文件哈希未变化。
- Git 暂存区、提交历史、CI artifact 和测试截图中没有真实样本或真值。
- 日志和报告敏感字段扫描为 0 命中。

### G1 文件与任务

- 所有文件都有明确处理路线或明确不支持原因。
- 不支持格式统一 4xx；模型/扫描服务不可用统一可恢复错误；无未知 500。
- 文件、导入、OCR 的任务状态、重试、取消、lease 和清理可验证。

### G2 数据正确性

- 所有确认记录通过项目、模板、字段、方向、Decimal 和状态校验。
- 汇总/日报/预算不与原始明细重复计入实际经营数据。
- L3 抽样记录和报表金额 100% 一致。
- 关键动作 audit/ledger 覆盖率 100%。

### G3 模型与人工复核

- OCR/AI 指标达到目标或有明确人工辅助降级声明。
- 未人工确认 OCR 自动入账数为 0。
- 低置信度、冲突、多页归属和复杂表格均可人工处理。
- 文本模型数字答案与工具结果 100% 一致。

### G4 工程回归

```powershell
cd backend
npm run build
npm test
npm run test:integration

cd ..
npm run build
npm run test:e2e
npm run check:hygiene
npm run model:check:all
```

模型真实验收另需：

```powershell
npm run model:resident
npm run model:status
```

只有 WSL2、Docker、GPU 和容器健康检查全部通过后，才执行真实模型批次。

每个批次报告至少包含：提交范围、样本族覆盖、通过/失败数量、性能指标、P0-P3 问题、修复文件、测试命令、未完成项和下一门禁。

## 14. 需要业务人员确认的事项

以下问题不能由模型或开发者自行猜测：

1. 考勤/司机/账单应按“明细行”还是“结算汇总”入账。
2. 预算、资金需求、备用金申请和实际核销的会计关系。
3. 理赔、冲销、退款和负数记录的方向与期间。
4. 哪些字段构成业务级重复，以及重复是阻断、告警还是允许版本替换。
5. 工资、银行卡、电话和地址的保留期限、可见角色和脱敏规则。
6. 35 页凭证应作为一个任务、按页拆分，还是按业务单据拆分。
7. OCR 最低可接受准确率和财务人员可接受的单据复核时长。

在这些答案给出前，可以继续完成扫描、兼容、状态机、安全、性能和人工辅助流程，但不能声称会计口径已最终验收。
