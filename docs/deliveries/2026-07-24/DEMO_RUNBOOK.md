# 周五现场演示 Runbook

目标时长：5-8 分钟。主线只演示 Excel、双财务审批、正式经营记录和老板审计快照；OCR 与 AI 叙述均不作为成功条件。

## 开场前准备

1. 确认 PostgreSQL 正在运行，`backend/.env.test` 的 `TEST_DATABASE_URL` 指向本机精确数据库 `finance_agent_test`。
2. 在仓库根目录运行：

```powershell
npm run demo:config:test
npm run demo:reset
npm run demo:verify
```

3. `demo:verify` 必须输出 `DEMO_VERIFY_OK`，并显示：两个 active 财务账号、一个 active 老板账号、`太和中转项目`、`运输费用模板`、3 行 fixture、合计 `13422.21`、AI/OCR 为 Mock、外部 Provider 为 disabled。
4. 分别启动 `npm run demo:api` 和 `npm run demo:web`，检查：

```powershell
Invoke-RestMethod http://127.0.0.1:3101/api/health/ready
```

5. 浏览器打开 `http://127.0.0.1:4173`。不要打开真实业务文件、真实模型控制台或 production 配置。

## 演示账号

以下都是现有非生产 seed 账号，密码均为 `123456`：

| 演示身份 | 用户名 | 用途 |
| --- | --- | --- |
| 财务 A | `finance` | 上传、选择区域、确认确定性映射；不能审批自己上传的任务 |

## 现场异常处理

- `demo:reset` 拒绝执行：不要改脚本绕过。检查 `NODE_ENV`、数据库主机和库名；只能使用本机 `finance_agent_test`。
- 端口占用：先识别占用 `3101`/`4173` 的本项目进程并正常结束，不要改成不受文档约束的随机服务地址。
- 页面没有最新任务：在“导入任务”刷新，并确认登录的是财务 B。不要让财务 A 继续批准。
- “重新校验”未通过：停止批准，保留页面错误并记录证据；不得现场删除 warning、降低断言或改库。
- Provider 不可用：主线使用确定性手工映射，不需要真实模型。不要临时启用外部 Provider。
- 现场时间不足：停在正式记录和确定性经营日报；不要为了展示效果临时点开未验收的 OCR/AI 路径。

## 演示后

关闭两个本地服务。下一次演示前再次运行 `npm run demo:reset` 和 `npm run demo:verify`，不要依赖上一次演示留下的数据。
