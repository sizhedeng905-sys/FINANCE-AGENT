## CR 在做什么

CR 是 Commit Review 的缩写。它不是新增一套业务模块，而是把一个可独立审查的风险或功能拆成小提交，并配一份说明目标、根因、改动、测试、限制和回滚方法的审查文档。审查者可从 [`docs/commit-reviews/README.md`](../commit-reviews/README.md) 按编号追踪。

| CR 范围 | 从功能上解决的问题 | 当前结论 |
| --- | --- | --- |
| CR001-CR010 | Excel 暂存隔离、发布完整性、Prompt 真执行、AI 系统登记、运行镜像供应链 | 工程门禁已建立；历史与当前证据分开保存 |
| CR011-CR016 | 周五 Excel 到经营报告 E2E、可重复 Demo 包、AI 建议进入财务草稿、人工决定留痕、第二财务确认、批准后定位本批记录 | 主展示链路完成；CR016 是报告编写时的 GitHub 远端 HEAD |
| CR017-CR023 | 把 Excel AI 输出绑定到生成时状态，统一 accept/edit/reject/ignore 真值，要求整批审核，绑定审核摘要，并使审核事实 append-only | P0/P1 数据完整性在本机完成攻击性验证 |
| CR024-CR032 | OCR 纠错版本前置、旧 AI 输出拒绝、原始 OCR 值保留、不可变审核证据、整批审核、批准摘要和财务审核工作台 | 仅合成图片/PDF 工程验证；真实 OCR 准确率仍需样本真值 |
| CR033-CR035 | ReportSnapshot 来源分页，以及财务到老板的 Narrative 文本复核工作流 | AI 只能建议文字，不能修改 Snapshot、Claim 或金额；正式复核政策仍待 owner 决策 |
| CR036-CR038 | Staging 参数化、target profile 失败关闭、17 项只读目标预检 | 本地/合成工程验证；未声称真实目标环境通过 |
| CR039 | 本次进度、Demo 和 GitHub 可审查性检查点 | 纯文档检查点，不改变运行时 |

## 周五可展示闭环

### 核心技术链路：10/10 已实现

10. 批准快照、审核摘要、来源哈希和本批记录可互相追踪。
