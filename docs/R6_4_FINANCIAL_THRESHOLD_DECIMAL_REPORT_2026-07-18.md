# R6.4 财务阈值 Decimal 精度修复与验收报告

日期：2026-07-18

分支：`agent/b8-stable-hardening`

问题：`R6-DECIMAL-001`

状态：`verified`（工程精度风险已关闭；正式币种、冲销和舍入口径仍受 H01/H02/H06 约束）

## 红灯复现

旧路径先把 `conditionJson.threshold` 转成 JavaScript `number`，再执行 `new Prisma.Decimal(number)`。实际复现：

```text
JSON input: 99999999999999.99
JS number:  99999999999999.98
```

修改前 API 会接受已经丢失 0.01 的 numeric 值并进入持久化/计算。定向红测同时证明规范字符串 `"0.1"` 被旧校验拒绝，无法建立无浮点往返的输入契约。

## 修复契约

| 项目 | `financial-threshold/1.0` 行为 |
| --- | --- |
| 新输入 | 只使用非负规范十进制字符串；接受 `0`、`0.1`、`0.01`，持久化为固定两位字符串 |
| 精度 | 最多两位小数；不四舍五入，第三位及以后直接拒绝 |
| 范围 | `0.00..999999999999.99`，与现有 `WorkOrder Decimal(14,2)` 的 12 位整数/2 位小数一致 |
| 格式 | 拒绝空值、空白、符号、负数、前导零、点前/点后缺位、科学计数法和非字符串对象 |
| 旧 numeric | 仅兼容 `0..999999999999` 的非负安全整数；立即规范化为字符串并返回/审计弃用警告 |
| 不安全 numeric | 小数、NaN/Infinity、非安全整数和越界值返回稳定 400，不尝试推断或舍入 |
| 计算 | 从规范字符串直接构造 `Prisma.Decimal`，使用 Decimal 比较 |
| 证据 | 规则条件、结果、异常、audit 和 ledger 保存规范阈值、Schema 版本、输入模式和兼容警告 |

JSON 解析后无法辨认一个安全整数 numeric 最初写作 `1000` 还是 `1e3`，因此旧 numeric 兼容按解析后的“安全整数”判定；新调用必须使用字符串，字符串 `"1e3"` 会被拒绝。该兼容只用于读取/迁移旧配置，不代表继续推荐 numeric API。

## 安全边界

- 本阶段没有制定币种换算、负数/冲销、税务、舍入或跨币种汇总规则；这些仍是 H01/H02/H06 的人工决策。
- 两位小数来自当前数据库金额列的既有 `Decimal(14,2)` 技术约束；额外小数位失败关闭，不擅自选择舍入方法。
- 最大值从旧服务的 `99999999999999.99` 收紧到可真实写入当前金额列的 `999999999999.99`，防止规则阈值超出被比较字段的可表示范围。
- 已持久化的安全整数 JSON number 仍可重放并写弃用证据；不安全旧值在执行时失败关闭，需财务管理员改为规范字符串。

## 数据库与回滚

本阶段不修改 Prisma schema，不新增 migration。Seed 中三个金额阈值和示例异常证据改为规范字符串；25 条 migration 的空库和升级路径继续通过。

回滚代码不会恢复已经规范化的字符串为 numeric，也不应删除历史执行证据。若需要回退服务，应先确认旧版本能读取字符串条件；当前建议只前滚修复，不回退到会发生浮点损失的实现。
